# 向量数据库不是知识管理

或者,至少在我看来,这是那个正在悄悄推高 AI 基础设施预算的架构错误。

![](./images/vector-databases-are-not-knowledge-management/01.png)
*"Image generated using OpenAI's DALL·E."*

有那么一类架构决定,做出来的当下看着合情合理,而几年之后,在产生它的约束消失之后,就变得不可辩护、甚至有些尴尬。

好,我知道你在想这都是在说 agile 嘛,但,据说那是一种"*流程*"而不是一种架构决定。况且,不幸的是,管理层还没意识到自己的尴尬(虽然几乎所有开发者都意识到了),不过我们也别在头几段就跑题跑太远。

我其实是在说 Retrieval Augmented Generation,在熟人那里也叫 RAG,它就是这类决定中的一个。它在它那个年代,是当 context window 还是四千 token 时让一个有用的对话式 AI 能在真实语料上工作的唯一办法,而且从那以后它一直是个还算合理的补丁。在我看来(标准免责声明,大家大概都会忽略),现在一百万 token 的 window 已经悄悄进入生产,它正在变成一个加在已经不再有意义存在的问题之上的基础设施负担。

## RAG 作为一种工程上的补偿

最初的想法本身够扎实。Embeddings 把意义压成一个向量,向量可以按相似度来检索,因此语料的 chunk 就可以基于模糊的语义匹配被检索出来,在用户的问题落到模型之前塞进它那点小小的工作记忆。整个 stack 必然是搭在一连串脆弱对齐之上的,每一环都得撑住,最后的答案才站得住脚。

首先,东西得对得上。Chunk 必须沿着 embedding 模型认为有意义的边界把概念整齐地切开,而它从来也切不太齐——哪怕你去精修、调参,甚至近亲繁殖式地用一个 LLM 来做这一意义化筛选。

第二,检索分数必须把真正包含答案的 chunk 浮上来,而不是那十几个仅仅"听起来像问题"的 chunk。是要程序化地。

第三,embeddings 必须捕捉到用户脑中所想的语义意图,而这本身就是一个有争议的可操作化。同样,哪怕你近亲繁殖式地用同一个(或别的、便宜点的)LLM 来帮忙做这个语义捕捉本身。这里也有一丝退化的味道,不过现在我们手头要忙的事情已经够多了。

最后,provenance 必须在 chunk 回到模型综合的旅程中存活下来,而它很少能干净地存活。Provenance 我也喜欢称为旁证式的,而它顶多和意义、embedding、检索一样好,甚至(再来一次)可能要近亲繁殖式地求助又一个 LLM 才能被有意义地塞进去。

整个架构是有条件的,前提是这一切在生产环境里同时成立,而失败模式会随着语料的体量按比例叠加放大。最常被错过的一点是:RAG 是对一个注定会消失的限制所做的工程补偿。

## Context window 等于"*吃掉*"了 RAG

请允许我为这样使用"*吃掉*"这个词道歉。我平时不会这么用,但它正好契合本文那种"破除炒作"的调子。无论如何,context window 已经做到了每一个观察曲线的技术人都会预期它做到的事——它在三年内增长了大约两个数量级。四千变八千,八千变三万二,三万二变十二万八,如今一百万已经悄悄可用而且还在涨。成本经济在最大的语料上仍然倾向于分块检索,但在小型和中型规模上已经彻底翻转。

那些一直在悄悄维护 wikilink 知识花园¹的人,[Obsidian](https://obsidian.md/) 加 markdown 的群体,外置前额叶皮层这门功夫,已经走到了一个让人不安的觉悟:LLM 可以直接读完整个 vault。

不需要 embedding,不需要检索,不需要打分,也不需要持续重建索引。Wikilink 就是知识图谱,context window 就是工作记忆,留下来的结构是一个人写出来并精心打理过的结构,而不是一个模型从分块文本对着一个本来就不是为此目的设计的向量空间度量推断出来的结构。

## Karpathy 收敛

我前阵子搭了自己的一套配置,工作假设是模型的 context 终将赶上我笔记的体量,后来才发现 Andrej Karpathy 独立地走到了几乎一模一样的架构²。

三层,Raw Sources、Wiki、Schema。三种操作,Ingest、Query、Lint。镜像几乎是一比一,镜像得有点尴尬。趋同进化总是一个好用的信号。两个人各自独立工作,因为同一个原因抵达同一个形状,这相当强烈地暗示了那个形状是对的,而那个替代品——基于语义抽取的 embedding 伪记忆——一直就只是对一个现在已被打补丁的缺陷的临时绕路。

## 向量数据库真正属于哪里

那么,向量数据库属于哪儿?属于它们最初待的地方,做它们一直擅长且任何被人精心打理过的 markdown vault 永远填补不了的事——在杂乱的、高基数的、无结构的事件数据里找相似品。

向量数据库不是记忆,它们是对几乎等同于噪声的东西做模式识别。异常关联、不规则事件流上的相似性发现、在一片自由文本日志行和被荒诞地打错标签的传感器事件³之上回答*"上一次它长这样的时候发生了什么"*这个问题。

这才是 embedding 相似度真正回报得起自己投入的地方,因为数据本质上就是没结构的,而问题真的是"给我找像这样的东西"而不是"用我灌进去的手册回答这个问题"。

我前阵子就是按这个前提搭过一个测试系统⁴,一个把 InfluxDB 用于原始时间序列信号、把 Chroma 用作语义召回层的小架构,给被检测到的异常打上丰富的元数据并对它们做 embedding,这样能源资产的运营人员就可以在多年来的历史事故上问"找相似品"的问题。

它是关于向量数据库该做什么的一个跑通了的样例,被部署在一个它们真的擅长的问题上。

## 一个能存活下来的分类法

从这个重新框定里浮现出来的分类法相当直接。

-   稳定的、人手精心打理的知识属于以 link 串起来的 markdown,直接读进模型的 context window。
-   不稳定的、网级别的无结构检索仍然受益于老的 RAG 模式,大致上。
    还有,
-   异常或事件流相似性发现是向量数据库的天然格子。
-   在稳定语料上的事实型问答经常更应该用 fine tuning 而不是检索本身。
-   还有,虽然我自己不用,我推测源代码搜索仍然是 embedding 一个有产出的小生境,至少目前如此。欢迎你的评论。

我相信最初的错误是把向量数据库当成记忆。它们不是记忆,而把它们当作记忆装的这一假装,过去近两年里一直在悄悄推高整个行业的成本、复杂度和失望。

-   知识想要的是被链接、被精心打理、被人写。
-   模式⁵想要的是被 embedded、被相似度评分、被机器检索。

把一类错认成另一类,就是这一代 AI 架构决定里写死的失败模式,它正在随着 context window 的限制退到后视镜里而缓慢地解开。

至少 context window 在变大,而多亏于此,也许 markdown 终于找到了它的位置。
