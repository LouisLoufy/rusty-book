# 1.1 什么是 Agent Harness

如果你最近半年用过 Claude Code、Cursor Agent、Cline、OpenAI Codex CLI 这类工具，大概率会对它们的能力感到惊讶：它们能读你的文件、改你的代码、跑你的命令、按需调用 MCP 工具、甚至打开浏览器自己点几下。但 LLM 本身只是一个**输入文字、输出文字**的函数，它不会读盘、不会写文件、不会发 HTTP 请求。

**「LLM 是大脑，Harness 是身体」**——这句话最近被各种 AI 工程师反复引用。这一章我们就从最基础的问题问起：那层让模型「能动手」的外壳到底是什么，它做了哪些 LLM 自己做不了的事，以及为什么它的设计决定了一个 Agent 产品的好坏。

## 一、Harness 这个词从哪来

`harness` 原本是「马具」，套在马身上把缰绳、嚼子、挽具固定起来，让骑手能控制方向、马的力气能传给车厢。AI 圈拿这个词来比喻**包在模型外面的那层运行时**，含义出奇地贴切：

- 马是力气源（模型生成 token 的能力）。
- 马具决定这股力气往哪去（工具调用路由、上下文剪裁、权限控制）。
- 没了马具，再壮的马也只能在原地打转。

学术圈也有人叫它 "agent scaffolding" 或 "agent runtime"，意思一样。本书统一用 Harness。

## 二、最小可运行的 Harness 是什么样

抛开所有花哨封装，一个 Harness 的核心循环可以浓缩到 10 行伪代码：

```python
def run_agent(user_prompt, tools):
    messages = [{"role": "user", "content": user_prompt}]
    while True:
        response = llm.complete(messages, tools=tools)
        messages.append(response)
        if response.stop_reason == "end_turn":
            return response.text
        for tool_call in response.tool_calls:
            result = tools[tool_call.name].run(**tool_call.args)
            messages.append({"role": "tool", "content": result})
```

就这么点东西——一个 `while True`，每轮问模型「下一步干啥」，模型说要调工具就帮它调、把结果塞回上下文，模型说「我说完了」就退出。所有市面上的 agent 产品，剥到骨头都是这个循环。它们之间真正的差距，全在循环周围那些「看上去不起眼」的细节。

## 三、Harness 要解决的五件事

### 1. 工具注册与调度

模型怎么知道有哪些工具能调？靠 Harness 在每次请求里把工具的 **JSON Schema** 喂进去，模型据此输出结构化的 `tool_call`。Harness 收到后要：
- **路由**：根据工具名找到对应的执行函数。
- **参数校验**：模型偶尔会写错参数；校验失败就把错误回传，让模型改而不是崩。
- **超时与重试**：长跑工具（编译、网络请求、子进程）需要超时；幂等工具可以静默重试。
- **并发控制**：同时调多个工具时，是顺序跑还是并发？读类工具可以并发，写类工具往往要排队。

### 2. 上下文与记忆管理

模型的 context window 总是有限的（Claude Opus 也就 200K token），但一个真实任务可能要看几十个文件、跑几十次工具。Harness 必须决定：
- **塞什么**：系统提示、工具结果、文件内容、对话历史，哪些进上下文、哪些丢掉？
- **怎么压缩**：超过阈值后，怎么把旧轮次摘要成短文本（compaction）？
- **怎么持久化**：哪些东西要写进长期记忆（memory 文件、向量库），下轮对话还能用？

Claude Code 之所以「长时间还能保持靠谱」，很大一部分功劳在 `/compact` 和它对长上下文的剪裁策略。

### 3. 权限与沙箱

模型一旦能写文件、能跑 shell，就需要**护栏**：
- 哪些路径只读、哪些可写？
- 哪些命令直接放行、哪些每次都要用户点同意、哪些彻底禁止（`rm -rf /`、`git push --force` ……）？
- 工具是不是要在沙箱（Docker、firejail、worktree）里跑，免得污染主环境？

这层做不好，要么 Agent 处处问用户、体验极差；要么 Agent 自作主张、闹出事故。Claude Code 的「permission mode」、Cursor 的「Agent Mode 白名单」、Cline 的 auto-approve 都是这件事的不同答案。

### 4. 产物与状态追踪

一次任务可能产出代码改动、新文件、命令输出、错误日志、未完事项。Harness 要：
- **结构化记录**这些产物（哪些文件改了、`git diff` 长什么样、跑了哪些命令）。
- 在 UI 上让用户**审计**（review）和**回滚**（undo）。
- 把「未完的事」写到 task 列表，下轮还能接着干。

### 5. 事件循环与人机协作

光会自己跑还不够，Harness 还要让人能随时**打断**、**纠正**、**追加需求**：
- 用户中途敲回车要插话怎么办？
- 模型跑到一半要问「这两种方案你选哪个」，怎么礼貌地暂停等输入？
- 后台任务完成了，怎么把通知送到模型的下一轮上下文里？

这一层的设计直接决定 Agent「像不像一个合作的同事」还是「像一台只会蛮干的机器」。

## 四、看几个主流 Harness 的取舍

| Harness                | 形态           | 工具策略                   | 上下文策略                   | 权限策略                          |
| ---------------------- | -------------- | -------------------------- | ---------------------------- | --------------------------------- |
| **Claude Code**        | CLI + 多客户端 | 内置 + Skills + MCP        | 显式 `/compact`、auto-memory | 三档 permission mode + 工具白名单 |
| **Cursor Agent**       | IDE 内嵌       | 文件读写为主 + Terminal    | IDE 提供的活跃文件优先       | Agent Mode 全自动 / 手动批准      |
| **OpenAI Agents SDK**  | SDK 框架       | 强类型 Tool 装饰器         | 由开发者自行管理             | 由开发者自行管理                  |
| **自研脚本（10 行）** | 一次性 Python  | hard-code 几个 function    | 全量塞、超了就报错           | 不设防、运行即同意                |

可以看到：越往左，Harness 帮你做的越多、灵活度越低；越往右，自由度越高、但你得自己实现上面那五件事。所以新人常见的认知错误是：「我自己 while-loop 也能写一个 Agent，为啥要用 Claude Code？」——能写出循环只是起点，把那五件事做扎实需要几千行代码和上百次的细节迭代。

## 五、为什么要专门学 Harness

业界关于「LLM 越来越强，Harness 就不重要了」的争论从没停过。但实际趋势恰恰相反：

- 模型变强 → 它愿意主动调更多工具 → 工具调度和权限控制更关键。
- 任务变长 → 上下文管理和长期记忆更关键。
- 多 Agent 协作流行 → 进程间通信、共享状态、并发控制更关键。

模型可能每半年换一代，但 Harness 这一层的设计思想是**沉淀性**的——你今天理解了 Claude Code 怎么剪裁上下文、怎么管 permissions、怎么调度 sub-agents，明天换成 GPT-6、Gemini 3 同样适用。

这本书会跟着 Claude Code 的开源 harness 一步步拆，把这五件事每件单独拿出来讲清楚。下一节我们先看**工具注册**：模型究竟是怎么「看到」可用工具的，JSON Schema 在这里扮演什么角色，以及为什么「工具描述写得好不好」直接影响模型的调用准确率。

---

> **本节小结**
>
> - Harness 是 LLM 之外的那层运行时，负责把「生成文字的能力」翻译成「在真实世界做事」的能力。
> - 它的最小骨架是一个 while 循环 + 工具调度，但真正的工程量在循环周围。
> - 五件大事：工具调度、上下文管理、权限沙箱、产物追踪、事件循环。
> - Harness 的设计思想比模型本身更稳定，值得专门学。
