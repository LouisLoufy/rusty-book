# 别做应激反应，要做主动回应：我们为"专注"重造了整套研发流程

## 创业公司文化建立在多任务之上，可人脑并不是这么造的。这是我们如何把 Ewake 的研发流程重新设计成一台死命守护专注力的机器

![一张极简风的工程示意图：一条传送带。左侧，一只机械机器人靴子把一团混乱的干扰物——Slack 图标、邮件信封，还有一个写着"HEY QUICK ONE！"的盒子——一脚踢下传送带。越过靴子，传送带继续平稳运行，上面只剩干净、均匀间隔、发着绿光的几何方块，代表专注而不被打断的工作。](https://cdn.jsdelivr.net/gh/beatai-org/beatai-assets@d862590ea9216ae63f9cd9b4163d944ec90098a3/ai-insights/2026-06/04/images/dont-react-respond-why-we-rebuilt-our-sdlc-for-focus/01.webp)

我是个反多任务的"卢德分子"。我不喜欢多任务，从来没喜欢过。可在我们这行，它是职业病，而 codegen 只让它来得更快（也更糟）。"持续多任务是一种糟糕的活法"——我这套理论只越来越坚定。

证据来得有点离谱：一条 Instagram reel。容我辩解一句，那是条引用出奇严谨的 reel。我中断了无脑刷屏，顺着它的研究追下去，一头扎进兔子洞，出来时手里攥着*实打实的证据*。

原来神经科学界的共识相当明确：多任务是个神话。人脑根本没有支撑它的构造。我们*以为*的多任务，其实只是快速的任务切换。

这事研究者早就在敲警钟：

-   [Mark 等 (2008)](https://dl.acm.org/doi/10.1145/1357054.1357072) 证明，在任务间来回横跳短期内或许让人感觉更快，但终归会通向倦怠。
-   就算让人完全自由地安排自己的日程，选择多任务的那批人，表现也明显不如逼自己一件一件顺序做的人（[Buser & Peter, 2012](https://www.cambridge.org/core/journals/experimental-economics/article/multitasking/0D8CADEA4C6F6ACDBB1E51615B5D67DA)）。
-   每次切换焦点，大脑都得卸下当前的认知框架、再加载一个新的。[Hasan (2024)](https://journals.lww.com/annals-of-medicine-and-surgery/fulltext/2024/11000/digital_multitasking_and_hyperactivity__unveiling.5.aspx) 发现，不断重新定位自己所需的心智负荷，可能吃掉**多达 40%** 的有效产出时间。更糟的是，这样切割资源会让工作记忆过载，直接导致心智疲劳和糟糕的决策。

既然它让我们更慢、把活干得更差，还实打实地榨干心理健康，那为什么我们还停不下来？

因为多巴胺的错觉。神经科学家 Daniel J. Levitin 在他的书 [*The Organized Mind*](https://www.daniellevitin.com/the-organized-mind) 里解释道：人脑天生就被设定成奖赏新奇。每切换一次上下文，大脑都因"挖到了新信息"而分泌一波多巴胺。你觉得自己很高产，因为你确实在不停"做事"。可现实是，你只是在奖赏大脑分心而已。

> **tl;dr**：Cian 读了 3（三）篇研究论文外加一本书的梗概，全都支持他那个观点——持续多任务又慢又伤身。

那个 40% 的数字着实吓了我一跳。其余的，不过是给我的看法做了背书。

带着"多巴胺错觉"这个视角去看我们自己的工程流程，我发现它无处不在。我们那套号称"敏捷"的创业文化，说穿了不过是给全团队、制度化的多任务找的借口。

## 我们一直在空转打滑

刚起步那会儿，我们干的是标准的"创业公司™️"那一套。本能告诉我们，从一个想法到拿到市场信号，走直线最快。我们一周才规划一次，几乎每个周期都被客户需求打断。事情开了头就一直做下去，直到冒出更要紧的事——而这经常发生。

我清楚记得九月里的某一周。整周我们都埋头在"真正"的活上，没时间做规划，结果周五带着一堆半成品想法走进规划会。那场会是我们第一次真正拷问这些想法，二十分钟不到就看出来它们根本没成型。于是周一只能重做一次规划。我们就这么原地空转。

这事的另一面——PR 队列——我们[此前写过](https://www.ewake.ai/blog/the-cost-of-cheap-code)：当评审节奏跟不上新代码涌入的速度，会发生什么。那是战术层。眼下这篇讲的是战略层，它在任何一行代码写出来之前就已经开始。单看个人，我们觉得自己很快；可摊到一个月，干成的事反而更少。代价只是没记在账上罢了。

## Shape Up Lite：先定"胃口"，超了就砍范围

我们四处淘方法论，最后看中了 Shape Up 对待范围（scope）的方式。它不去估算一件事要花多久（在 codegen 工具当道的今天，这种估算既别扭又困难），而是先定一个"胃口"（appetite）——"作为一个团队，我们愿意在这个项目上投入这么多时间"。一旦撞到胃口上限，要么砍范围，要么加胃口。关键在于：我们就此做出一个决定。

范围在动工前用一份叫 one-pager 的文档定好。这份文档会明确写出胃口、哪些在范围内、（最关键的）哪些在范围之外，以及哪些风险因素可能让项目撑爆胃口。

每个项目都从一份 one-pager 起步，我们提前定义范围的本事也越来越好。但当时我们还没准备好彻底切换到 Shape Up 那套"主动周期 + 冷却期"的完整模型，于是一边沿用着每两周一次的常规 sprint 规划，一边并行试水。

另一个改动是关于技术设计的思路。我们本想并行推进，结果却各自筑起了孤岛——总是在 PR 里才开始讨论设计，代码都写完了，才回过头争论底层路子对不对。后来我们开始强调：动代码之前，先把设计落在纸上。

## 把工作向左移：越早定，越省时

道理很简单：

> 花在范围上的时间，省下更多设计时间；花在设计上的时间，又省下更多代码评审和实现的时间。

效率提升立竿见影：

-   过去在 PR 里要吵一天的设计分歧，挪到设计评审阶段一小时就理清了。
-   漏掉的需求在范围评审时就被抓住，不必事后再硬塞回去。
-   烂特性还没起飞就被砍掉，而不是等它的 PR 被卡住。

codegen 让这一切变得更要紧。[很久很久以前](https://en.wiktionary.org/wiki/fad%C3%B3_fad%C3%B3)，一个工程师实现一个特性要花三天，我们还有时间边做边给设计做方向性的微调。而 Claude Code 是"发射后不管"——你要是把它指错了方向，它会兴高采烈地在转眼间给你实现出一个糟糕透顶的设计。如今生成代码几乎是免费的，人的架构思考反倒成了整个研发流程里最值钱的东西。动手之前就清楚自己到底在造什么，已经不是可选项，它就是工作本身。

有那么几周，这感觉像一场顿悟。范围问题修好了，PR 队列重新喘上了气，工程师感觉自己对技术选型真正握有了主导权。

## 还差得远

我们仍在"真正"的活的缝隙里硬挤 one-pager，很少有时间好好准备。结果常常是随手给一张内容稀薄的工单丢一个"一周胃口"，只为让事情转起来——这逼得我们在实时推进中临时摸索范围，而不是真正地去塑形它。

因为 one-pager 是一个接一个串行写的，工程师做完手头项目，往往发现前面没有塑好形的活在等着。于是他们急刹车、起草 one-pager、等异步评审，等待期间顺手捡些零碎的"快速见效"小活。那些真正要紧的事，势头一次次熄火。

最后，对于周期中途调整优先级，我们还是缺一套像样的机制。活进来了，我们要么扑上去做，要么塞进 Linear，盼着以后能轮到它。

把 Shape Up 做了一半，我们造出来个"叉勺"——喝汤是比叉子强，可我们想要的不止于此。

## 干脆全押上

我们的解法是彻底承诺 Shape Up：一段火力全开的主动周期（active cycle），紧接一段用于维护、规划和准备的冷却期（cooldown）。Basecamp 默认的节奏是六周开工、两周冷却。这个比例对他们也许管用，但对一家 8 人的创业公司，6 周锁太久了。我们对半砍，改成三周主动周期 + 一周冷却。

冷却那一周不是休假，是为下一程做准备。工程师在这周扮演"塑形者"（Shaper），跟创始人异步打磨下一个主动周期的 one-pager，把胃口辩清、敲定。到冷却期结束，我们手里就攒下了一批塑好形、可以开打的"赌注"（也就是项目）。

每个主动周期前的规划会，已经缩成一场简短的*下注会*（Betting Table）：摊开塑好形的 one-pager，决定接下来三周给哪些赌注投钱。一散会，团队整整一个月没有障碍——主动周期里没人再需要起草 one-pager，也没人再为范围争论。我们只管造。

## 这是一场零和游戏

读到这儿你大概在想：三周对一家创业公司来说，是不是锁得太死了？没错。让它行得通的唯一办法，是把这个周期当成一场零和游戏。

要是周期中途来了件天大的要紧事，我们当然该做。但必须想明白：做这件，就意味着周期里另一件得掉队。于是话题从"我们能做这个吗？"（答案几乎永远是"技术上能"）变成了"我们该用这件去换掉那件吗？"。

认知科学在这里短暂地回归一下：打断对发起者是免费的，成本却由整个团队承担。我们管这叫**上下文税**（Context Tax），而这个周期机制让它显形。我们不是要躲着不交——有时新来的事确实更重要。我们只是确保自己是有意识地在交，并且换来的东西配得上这个价。

## 实际跑起来是什么样

1.  **新想法**：想法可能来自和客户的对话、市场调研，或我们这支有才又谦逊的团队天生的创造力。它先进入 Notion 里的想法池，等待被塑形。
2.  **用 One-Pager 定范围**：冷却周里，一名工程师领走某个想法，跟创始人和团队异步协作，敲定胃口、严格划定范围、识别风险因素。这从一小时到大约一天不等。范围一旦钉死，就可以进入下一步。
3.  **下注会**：摊开所有塑好形的 one-pager，决定把哪些拉进下一个周期。没能进这个周期的，先停泊一边，下个冷却周再回来看。
4.  **主动周期**：这是轻松的部分——全团队要做的就是埋头猛造三周。没有范围争论，没有规划。胃口用光了，要么砍范围，要么延长项目、代价是牺牲周期里靠后的某件事。
5.  **应对打断**：周期中途有新东西进来。我们停一下，问自己："如果想做这个，那要砍掉什么？"把上下文税摆到台面上明说，然后有意识地做决定——要么新事胜出、别的被踢掉，要么它等到下一场下注会。

## 跑起来的感受

第一个冷却周听上去像个轻松的礼拜：没有特性开发，没有 sprint 目标。能有多忙？

事实证明，相当忙。一直想修的 bug、一直绕着走的技术债、团队惦记了好久的开发体验改进。那一周我们照样发了三个 release。其中有种奇妙的、带着负罪感的快乐——我们没造任何新东西，却在为下一个周期打地基。

主动周期和过去那种"边走边掉东西"的状态完全不同。我们有了计划，每个人都清楚自己在做什么、从哪儿下手，第一天就能扎进去。它让人觉得平静、从容。算不上完美，但也绝不慌乱。

我们更快了，因为不再走走停停。活干得更好，因为人们能把一整个问题在脑子里盘上超过二十分钟不被打断。决策更好，因为它们是在我们被逼着做之前就做好的。

我们并没有变得不忙，也没挖出什么隐藏的额外工时。我们只是不再稀里糊涂地交上下文税——而当我们确实要交时（因为有时不得不交），是有意识地交的。

## 还没搞定的难题

一个词：拿捏分寸（calibration）。

零和规则逼我们对范围诚实，但我们并不总能拿捏准。有些在下注会上看着很紧凑的 one-pager，做起来才发现比表面宽得多。我们在慢慢变好。

另一件难事是判断紧急程度。这套周期逼出了一条清晰的界线：什么*是*紧急，什么只是*感觉*紧急。这是个需要拍板的判断，我们偶尔会判错。人生本就如此。

细节我们还在摸索，但大方向看起来是奏效的。那种慌乱的感觉没了，活也推得快。这就是全部的意义所在。

真正变了的，不是我们交付什么，而是那种"想到一半、且笃定没人会把你拽出来"的踏实感。

我们照旧拼命干、不停发版、被各种意外球砸中。但有一份大家共同参与制定的计划摆在那儿。紧急的事砸下来，我们知道怎么接。周期结束时，总有件完成的东西可以指给人看。

说实话，这感觉爽——极了。
